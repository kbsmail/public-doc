조치 근거 (보강)

#7 TLS 1.3 — TLS 1.3은 Exchange 기능이 아니라 새 OS(WS2022+)의 Schannel 기능으로, SE 전환이 새 OS와 함께 오면서 활성화되었다. 따라서 OS(Schannel) 레벨에서 Disable 처리하였다.

#11 확장된 보호(EP) — EP는 인증을 외부 TLS 채널에 묶는 채널 바인딩 토큰(CBT) 방식인데, SSL 오프로딩은 TLS가 LB에서 종료되어 CBT가 불일치(중간자로 간주)한다. 즉 EP와 SSL 오프로딩은 구조적으로 비호환이며, 오프로딩을 유지하는 한 EP 비활성 유지는 선택이 아니라 필수이다.

부록 B. SK-MAIL-N10 — /mapi sslFlags 특이사항 (유일한 예외)

SK-MAIL-N01 ~ N15 중 N10에서만 발생. Default Web Site/mapi access.sslFlags가 264 → 8로 자동 변경되었으며(나머지는 264 유지), 미조치 8로 그대로 두었다.

미조치해도 영향이 없는 이유

1. 8도 SSL 필수 유지 — 8 = Ssl, 264 = Ssl + Ssl128. 두 값 모두 Ssl 비트를 포함하므로 /mapi는 N10에서도 HTTPS만 허용한다(SSL 강제 불변).

2. 빠진 것은 Ssl128(256) 한 비트 — "128비트 미만 cipher 거부"를 강제하는 플래그이다.

3. 현대 OS엔 128비트 미만 cipher가 없음 — export급(40·56비트) cipher는 이미 제거되어, 실제 협상 cipher는 264든 8이든 모두 128비트 이상으로 동일하다.

4. SSL 오프로딩 구조 — 서버는 LB하고만 TLS를 협상하므로 영향 범위가 LB 한 구간으로 한정되고, 그마저도 cipher 차이가 없다.

→ SSL 요구·암호화 강도·클라이언트 동작 모두 동일하여 실질 영향 없음. (16대 값 통일을 원할 경우 Ssl,Ssl128(264)로 원복 가능)
