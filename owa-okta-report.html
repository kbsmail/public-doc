OWA Okta 로그인 간헐적 지연 — 원인 분석 및 조치

대상 시스템그룹Portal 통합메일 OWA (SK-MAIL-N01 ~ N16) 문서번호—
작성일2026-06-25 작성자강범석 / 믹시스템
관련 작업OWA Okta(WS-Fed) 인증 전환 상태조치 완료 · 효과 확인

1. 요약

OWA의 Okta 인증 전환 후, 사용자 로그인 마지막 단계에서 간헐적으로 약 13초 지연이 발생하였다. 원인은 폐쇄망 서버가 ADFS 적용을 위해 등록한 Okta 서명 인증서(okta.cert) 검증 과정에서 Microsoft 신뢰/거부 목록(CTL)을 외부(Windows Update)에서 받으려다 도달 불가로 약 15초 타임아웃한 것이다. 해당 외부 조회 시도를 비활성화하는 레지스트리 설정을 전 서버에 적용하여 응답 시간이 1초 미만으로 정상화되었다.

3. 원인 분석

3.1 결합된 3대 요인

요인설명
① 폐쇄망 환경서버가 외부(ctldl.windowsupdate.com)로 통신 불가
② ADFS/Okta 전환전환 후 로그인마다, 등록한 Okta 서명 인증서(okta.cert)로 서명된 토큰의 인증서 체인 검증 수행(기존 폼 인증에는 없던 동작)
③ CTL 자동 갱신검증 시 Windows가 신뢰/거부 목록(CTL) 3종(authrootstl·disallowedcertstl·pinrulesstl)을 외부에서 받으려 함

3.2 동작 메커니즘 (CryptoAPI 네트워크 조회 + 도달 불가 URL 처리)

인증서 검증 시 CryptoAPI는 폐기/목록 객체를 네트워크로 받는다. MS 문서가 명시하는 동작은 다음과 같다.

  • 디스크 캐시에 유효 객체가 없으면 백그라운드 스레드로 다운로드 시작, 호출 스레드는 최대 15초 대기.
  • 15초 안에 못 받으면 서버를 offline으로 표시(다운로드는 백그라운드 계속). → 이 한 번이 사용자 약 13초 지연.
  • 폐쇄망처럼 도달 불가한 URL은 "도달 불가 URL 목록"(프로세스별 관리)에 등록.
  • 다음 시도 허용 시각 전까지의 모든 요청은 즉시 offline 오류로 실패(네트워크 시도 안 함 = 빠르게 통과).
  • 허용 시각에 재시도해서 또 실패하면 대기 시간을 늘림(증가 방식은 문서 비명시), 최대 30분. 성공하면 15초로 리셋(폐쇄망은 성공이 없어 미해당).
[근거] Microsoft Learn — How Certificate Revocation Works (Network Retrieval and Caching)
"If the object takes longer than 15 seconds to download, then CryptoAPI will report the server as offline, even as the retrieval continues in the background."
"The unreachable URL is added to a list of unreachable URLs. The list of unreachable URLs is maintained on a per process basis. ... any attempt to retrieve the object will fail immediately with an offline error. ... CryptoAPI will extend the time it will wait until another attempt is processed. The maximum time that it will wait is 30 minutes. If the attempted connection is successful, then the interval is reset to 15 seconds."
출처: learn.microsoft.com/.../ee619754 (How Certificate Revocation Works)

3.3 왜 "가끔" 발생했고, 30분보다 훨씬 잦았는가

OFFLINE 캐시가 유효한 동안의 검증은 외부 조회를 건너뛰어 즉시 통과(정상)되고, 재시도가 발생하는 시점의 검증만 약 13초 지연된다. 그래서 재시도 시점에 로그인한 경우에만 느렸고 그 사이는 정상이었던 것이 — "간헐적"으로 보인 이유다.

지연 재발이 "30분마다"보다 훨씬 잦았던 이유는, 문서의 30분은 "단일 URL·단일 프로세스가 오래 연속 실패했을 때 도달하는 상한"일 뿐, 실제로는 독립 타이머가 다수 동시에 돌기 때문이다(아래는 검증된 사실에 근거한 설명):

  • 목록은 프로세스별(문서 명시). OWA·ECP 앱풀의 워커 프로세스가 여럿이고, 앱풀 재활용/유휴 종료 시마다 목록이 초기화되어 다시 15초부터(짧은 간격) 시작.
  • URL이 다수 — CTL 3종 + 인증서 체인의 CRL/OCSP가 각자 독립 백오프 타이머를 가짐.
  • 서버 16대 + LB 분산 — 서버·프로세스마다 상태가 따로. 매번 다른 백엔드로 분산될 수 있음.

→ 체감 재발은 이 수많은 독립 타이머의 합집합이며, 다수가 아직 짧은 초기 간격이거나 재활용으로 리셋되므로 단일 "30분"보다 잦게 나타난다. 정확한 재발 주기는 단일 값으로 도출되지 않으며 본 보고서는 단정하지 않는다. 실제 발생 시각은 3.4의 캐시 쓰여진 시각(실측)으로만 확인한다. (테스트는 매번 새 창=새 인증으로 수행하여 브라우저 세션 영향 배제)

4. 조치 내역

외부 목록 자동 갱신 시도를 비활성화하는 레지스트리를 전 서버(16대)에 적용(폐쇄망 표준 대응).

:: 관리자 권한 CMD (서버별 적용)
reg add "HKLM\SOFTWARE\Policies\Microsoft\SystemCertificates\AuthRoot" /v DisableRootAutoUpdate /t REG_DWORD /d 1 /f

:: 반영
gpupdate /force  &  iisreset

:: 적용 확인 (0x1)
reg query "HKLM\SOFTWARE\Policies\Microsoft\SystemCertificates\AuthRoot" /v DisableRootAutoUpdate
항목내용
적용 범위서버(컴퓨터) 단위 — HKLM / 메일 서버 16대 전체
효과인증서 검증이 로컬 목록만 사용 → 외부 조회 시도 제거 → 타임아웃 소멸

5. 조치 결과

조치 후 동일 명령(certutil … -verify) 소요 시간이 1초 미만으로 단축되었으며, 사용자 로그인 지연 현상이 재현되지 않음을 확인하였다.

6. 재발 방지 및 향후 관리

  • 신규 도입 서버에도 동일 설정을 표준 적용(GPO 배포 권장).
  • 자동 갱신 비활성화에 따라 신뢰 루트·거부 목록은 수동(오프라인 SST/GPO/WSUS)으로 주기적 갱신.
  • 설정은 가역적이며, 값 삭제 또는 0 변경으로 원복 가능.
