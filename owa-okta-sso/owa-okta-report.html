OWA Okta 로그인 간헐적 지연 — 원인 분석 및 조치

대상 시스템그룹Portal 통합메일 OWA (SK-MAIL-N01 ~ N16) 문서번호—
작성일2026-06-25 작성자강범석 / 믹시스템
관련 작업OWA Okta(WS-Fed) 인증 전환 상태조치 완료 · 효과 확인

1. 요약 (Executive Summary)

OWA의 Okta 인증 전환 후, 사용자 로그인 마지막 단계에서 간헐적으로 약 13초 지연이 발생하였다. 원인은 폐쇄망 서버가 ADFS 적용을 위해 등록한 Okta 서명 인증서(okta.cert) 검증 과정에서 Microsoft 신뢰 인증서 목록을 외부(Windows Update)에서 갱신하려다 응답을 받지 못해 약 15초간 대기한 것으로 확인되었다. 해당 외부 갱신 시도를 비활성화하는 레지스트리 설정 적용으로 지연이 해소되었으며, 조치 후 응답 시간이 1초 미만으로 정상화되었다.

2. 발생 현상 및 영향

구분내용
현상Okta 로그인 후 mail.sk.com 복귀 단계에서 화면 멈춤(약 13초)
발생 양상상시가 아닌 간헐적 발생 (대부분 정상 응답)
영향 범위OWA 웹 로그인 사용자 (체감 지연). 메일 송수신 등 기능 장애는 없음
심각도중 (서비스 중단 아님, 사용자 경험 저하)

3. 원인 분석

세 가지 요인이 결합되어 발생하였다.

요인설명
① 폐쇄망 환경서버가 외부(ctldl.windowsupdate.com)로 통신 불가
② ADFS/Okta 전환전환 후 로그인마다, 본 작업에서 ADFS 적용을 위해 등록한 Okta 서명 인증서(okta.cert) 로 서명된 토큰의 인증서 체인 검증을 수행(기존 폼 인증에는 없던 동작)
③ 인증서 목록 자동 갱신인증서 검증 시 Windows가 신뢰/거부 목록(CTL)을 외부에서 자동 갱신하려는 기본 동작

왜 "가끔"만 발생했는가 (간헐성의 핵심 — 부정 캐시)

같은 인증서 검증인데도 어떤 로그인은 정상이고 어떤 로그인만 13초가 걸린 이유는, 외부 목록(CTL) 조회가 성공했을 때와 실패했을 때 캐시 수명이 완전히 다르기 때문이다.

상황캐시 동작지연
인터넷 가능(정상망)1회 성공 후 장기 캐시(수일 단위)거의 없음
폐쇄망(본 환경)매번 실패 → "실패했음" 상태를 짧게만(분 단위) 기억 = 부정 캐시(negative cache)짧은 주기로 반복

폐쇄망에서는 다운로드가 매번 실패하고, 그 부정 캐시는 일정 시간(관찰값: 수 분~십수 분) 만 유지된다. 그 결과:

  • 부정 캐시가 살아있는 동안 들어온 로그인 → "방금 실패했으니 다시 안 나감" → 즉시 통과(정상 속도)
  • 부정 캐시가 만료된 직후 들어온 로그인 → 죽은 주소(ctldl…)를 다시 조회 → 약 13초 대기 후 통과

따라서 부정 캐시 주기에 한 번꼴로 느린 로그인이 나타나고 그 사이 로그인은 정상이었던 것이 — 바로 "간헐적"으로 보인 이유다. (정확한 주기는 환경별로 다르며 별첨 방법으로 실측)

본 현상은 브라우저 캐시와 무관하다. 테스트는 매번 새 창(새 인증)으로 수행하여 브라우저 세션 영향이 배제되었으며, 지연은 전적으로 서버 측 인증서 검증의 재시도 타이밍(부정 캐시 만료 주기)에서 비롯된다.

원인 확인 및 검증 데이터

확인 항목방법결과 (증거)
1회 조회 타임아웃certutil -urlfetch -verify (온라인 강제)매회 약 13초 · 100% 재현
캐시 실패 상태certutil -urlcache "…authrootstl.cab"**** OFFLINE **** (다운로드 실패 캐시)
재시도(부정 캐시) 주기CryptnetUrlCache 메타데이터 LastWriteTime 관찰관찰값: 수 분~십수 분 간격 갱신 (환경별 상이)

검증 대상 인증서는 본 ADFS 전환을 위해 등록한 Okta 서명 인증서이다 (okta.cert / 자체서명 / Thumbprint 8FEAD90D3CEB90E62B6F46EA6A871C6391103754, Set-OrganizationConfig -AdfsSignCertificateThumbprints 로 등록). 이 인증서 자체에는 조회용 주소(CDP/OCSP)가 전혀 없음에도 검증이 지연된 점에서, 원인이 "인증서 폐기 조회"가 아닌 OS의 신뢰목록(CTL) 자동 갱신(외부 통신)임을 특정하였다. -urlfetch 는 캐시를 무시하고 온라인 조회를 강제하므로 폐쇄망에서 100% 타임아웃이 재현된다.

실측 증거 — 부정 캐시 상태와 "느렸던 시각"

ADFS(Okta) 로그인 시 서버는 등록한 서명 인증서의 체인을 검증하며, 이때 Windows는 신뢰/거부 목록(CTL) 3종(authrootstl · disallowedcertstl · pinrulesstl)을 외부에서 조회·갱신하려 한다. 폐쇄망에서는 3종 모두 다운로드에 실패하여 OFFLINE 상태의 부정 캐시로 보관된다.

캡처 ① 부정 캐시 상태 — 3종 CTL 모두 OFFLINE (SK-MAIL-N03 실측)

PS> 'authrootstl','disallowedcertstl','pinrulesstl' | % { certutil -urlcache "http://ctldl.windowsupdate.com/.../$_.cab" }

===== authrootstl =====
http://ctldl.windowsupdate.com/msdownload/update/v3/static/trustedr/en/authrootstl.cab
WinHttp 캐시 항목: 1
****  OFFLINE  ****
===== disallowedcertstl =====
****  OFFLINE  ****
===== pinrulesstl =====
****  OFFLINE  ****

캡처 ② 캐시가 "쓰여진(=재시도된) 시각" — 이 시각이 곧 "느렸던 로그인" 시각

CTL쓰여진 시각프로필
pinrulesstl2026-06-25 09:49:33NetworkService
disallowedcertstl2026-06-25 09:49:21NetworkService
pinrulesstl2026-06-25 09:47:56systemprofile
disallowedcertstl2026-06-25 09:45:40systemprofile
authrootstl2026-06-25 09:07:09NetworkService
authrootstl2026-06-25 08:09:49systemprofile

(가) 현재처럼 OFFLINE 부정 캐시가 남아 있는 상태는 "이미 실패를 알고 있어" 외부 조회를 건너뛰므로 로그인이 빠르게 통과된다.
(나) 위 "쓰여진 시각"이 곧 부정 캐시가 만료되어 재시도(=약 13초 지연)가 발생한 시각이며, 그 사이 시각의 로그인은 정상이었다. (검증마다 확인되는 거부목록 disallowedcertstl·pinrulesstl 이 09:45~09:49에 갱신 → 그 시점이 느린 로그인. authrootstl 은 더 뜸하게 갱신.)
(다) 정확한 부정 캐시 수명(주기)은 Microsoft 공식 문서에서 명시값을 찾지 못했다. 다만 "언제 느렸는지"는 위 캐시 쓰여진 시각으로 확인 가능하며, 동일 명령을 수 분 간격으로 두 번 캡처해 시각이 바뀌는 간격을 보면 재시도 주기를 실측할 수 있다.

4. 조치 내역

외부 목록 자동 갱신 시도를 비활성화하는 레지스트리를 전 서버(16대)에 적용하였다. (폐쇄망 표준 대응)

:: 관리자 권한 CMD (서버별 적용)
reg add "HKLM\SOFTWARE\Policies\Microsoft\SystemCertificates\AuthRoot" ^
  /v DisableRootAutoUpdate /t REG_DWORD /d 1 /f

:: 반영
gpupdate /force  &  iisreset

:: 적용 확인 (0x1)
reg query "HKLM\SOFTWARE\Policies\Microsoft\SystemCertificates\AuthRoot" /v DisableRootAutoUpdate
항목내용
적용 범위서버(컴퓨터) 단위 — HKLM / 메일 서버 16대 전체
효과인증서 검증이 로컬 목록만 사용 → 외부 대기 제거

6. 재발 방지 및 향후 관리

  • 신규 도입 서버에도 동일 레지스트리 설정을 표준 적용 (GPO 배포 권장)
  • 자동 갱신을 비활성화하였으므로 신뢰 루트 인증서는 수동(또는 GPO/WSUS)으로 관리
  • 설정은 가역적이며, 필요 시 값 삭제 또는 0 변경으로 원복 가능
[참고] 비전문가용 비유 — 손님(로그인)이 신분증(인증서)을 내면 직원(서버)은 "위조·정지 명단"에 없는지 확인한다. 그 명단을 가끔 본부(인터넷)에서 갱신하려 하는데, 폐쇄망은 전화선이 끊겨 있어 응답을 15초 기다린 뒤 통과시켰던 것이다. "본부에 전화하는 규칙"을 끄면 책상 위 사본(로컬 목록)만 보고 즉시 통과한다.
